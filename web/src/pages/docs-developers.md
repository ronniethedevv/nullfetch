# Use APIs from the NullFetch marketplace

> Developer docs · consumer guide
>
> Source: https://nullfetch.vercel.app/docs/developers

Set up a wallet, register for a service, and call its API from your
code. Your API keys are generated by your browser, never stored at the
provider or the marketplace. Possession is proven on-chain; the rest is
a normal HTTP call.

---

## 01 · Overview

As a developer using NullFetch, you do four things over the lifetime of
a subscription:

1. **Register** for a service once. Your browser generates a random
   32-byte API key, encrypts it client-side, and submits only the
   ciphertext on-chain. The plaintext is saved locally — by you, in
   your browser's storage.
2. **Attest** when you want to use the API. Re-encrypt your key, prove
   on-chain that it matches the ciphertext on file, and publish the
   result. Valid for 1 hour by default.
3. **Call the API** using a SIWE-style wallet signature per request.
   The provider's server reads your attestation and gates the
   response. No gas per call.
4. **Refresh** the attestation when it expires (or rotate your key if
   you want a new one).

The marketplace contract and the provider both see only ciphertext.
Your plaintext key exists only on the device that generated it.

## 03 · Register for your first service

### 1. Find a service

Open `/browse`. Every listed service is visible without a wallet. Click
any card to see its provider, endpoint, category, and current subscriber
count.

### 2. Click "Register for this service"

You'll land on `/developer/register/N`. The page shows the registration
fee (0.0003 Sepolia ETH) and the service's current status.

### 3. Watch what happens before MetaMask prompts

Your browser:

- Generates 32 random bytes with `crypto.getRandomValues`
- Computes `keccak256`, splits the hash into hi/lo halves
- Encrypts both halves locally using Zama's WASM SDK and produces a
  zero-knowledge proof of plaintext knowledge

Only the ciphertext + proof go to MetaMask. The plaintext key never
leaves your machine.

### 4. Confirm in MetaMask + pay the fee

You'll see a transaction for 0.0003 ETH. Confirm. Wait ~15 seconds for
the tx to mine.

### 5. Save your key — one-time reveal

The success page shows your plaintext API key behind a "Reveal" button.
Click it, copy the key somewhere you trust (password manager, secure
notes), and check the acknowledgement box.

The key is also auto-saved to this browser's local storage, encrypted
with a wallet-signature-derived key (see the threat-model doc for
details). But you should keep a backup outside the browser too.

> If you lose the plaintext key, the provider must revoke your
> subscription and you must register again (paying the fee again).
> There's no recovery.

## 04 · Run an attestation

Before you can call a NullFetch-gated API, you need a **fresh
attestation** on-chain. This is what proves to the provider that you
currently hold the right key, without ever showing them the key.

### 1. Open the use page

Go to `/developer/service/N` for the service you registered for. Your
key auto-fills from local storage (you may be prompted to unlock with
a wallet signature).

### 2. Click "run on-chain attestation"

This is a two-transaction flow on Sepolia:

1. `verifyAndAttest` — re-encrypts your key, compares it to the
   on-chain ciphertext under FHE, and marks the result publicly
   decryptable. Costs ~300K gas.
2. `submitAttestation` — Zama's KMS decrypts the comparison result and
   writes the cleartext boolean to the contract. Costs ~250K gas.

Both txs require a MetaMask confirmation. Total elapsed time: ~30
seconds. After both mine, you have a fresh attestation valid for 1
hour.

The attestation is keyed by your wallet, not by the API itself. One
attestation per (wallet, service); refreshing overwrites the previous
one.

## 05 · Call the API from code

Once you have a fresh attestation, each API call is a three-step
dance: fetch a challenge, sign it, send the headers. No gas, no
contract interaction.

> **Two things to know before you copy this code:**
>
> 1. Your wallet must be on **Sepolia** (chainId `11155111`). The
>    marketplace contract lives at `0x77CD…3875` on Sepolia only —
>    sign on the wrong chain and the attestation lookup hits a
>    contract that doesn't exist.
> 2. Sign the `message` field returned by `/challenge` **verbatim**.
>    Don't rebuild it from the `nonce` and `expiresAt` fields. The
>    server string-compares against the exact message it generated,
>    so any whitespace difference or line reordering breaks the
>    signature recovery.

```ts
import { BrowserProvider, getAddress } from 'ethers';

// the provider's deployed API base
const API = 'https://provider.example.dev';
const SERVICE_ID = 3;

async function call() {
  const provider = new BrowserProvider(window.ethereum);
  const signer = await provider.getSigner();

  // EIP-55 checksum — MetaMask returns lowercase, server rejects it
  const wallet = getAddress(await signer.getAddress());

  // 1. fetch a single-use challenge
  const cr = await fetch(`${API}/challenge?wallet=${wallet}`, {
    method: 'POST',
  });
  const { nonce, message } = await cr.json();

  // 2. sign the returned 'message' VERBATIM — don't rebuild it
  const signature = await signer.signMessage(message);

  // 3. call the gated endpoint
  const r = await fetch(`${API}/api/service/${SERVICE_ID}`, {
    headers: {
      'X-Wallet-Address': wallet,
      'X-Auth-Nonce': nonce,
      'X-Wallet-Signature': signature,
    },
  });

  // 4. ALWAYS check r.ok — a 401 body is valid JSON with the same
  //    Content-Type as success. Don't let it slip through silently.
  const body = await r.json();
  if (!r.ok) {
    throw new Error(`${body.error}: ${body.detail}`);
  }
  return body;
}
```

The challenge is single-use and expires after 2 minutes. You fetch a
fresh one for every call. MetaMask shows a signature popup each time
— there's no auto-approve setting for `personal_sign`; expect one
click per call.

### What you'll get back

The gate just authenticates; it doesn't reshape the response. The
shape is whatever the provider returns.

The reference NullFetch demo API and the example
[quote-api](https://github.com/ronniethedevv/quote-api) provider both
wrap their payloads as:

```json
{
  "authenticated": true,
  "wallet": "0x...",
  "service": { "id": "3", "name": "..." },
  "response": { "...the provider's actual data..." },
  "attestationExpiresInSeconds": 3580
}
```

That's a convention the SDK encourages but does not enforce —
providers can return any shape they want. Check the provider's own
docs for what their endpoint returns. If they follow the convention,
your data lives at `body.response`; if they don't, it's at the root.

### Handle errors

A 401 body has the same JSON shape and Content-Type as a 200 body —
`await r.json()` happily returns it as structured data. Don't trust
it without checking `r.ok`:

```ts
const body = await r.json();
if (!r.ok) {
  throw new Error(`${body.error}: ${body.detail}`);
}
return body;
```

See [§10](#10--error-codes-you-might-see) for every error code, what
causes it, and how to recover.

> The exact endpoint path (`/api/service/:id`) is a convention the
> NullFetch reference uses, but providers can mount their gated
> endpoint anywhere. Check the provider's docs for the actual path.

## 06 · Calling from a server / CI

For server-side or CI use, swap `BrowserProvider` for an `ethers.Wallet`
with a private key:

```ts
import 'dotenv/config';
import { Wallet } from 'ethers';

const wallet = new Wallet(process.env.SEPOLIA_PRIVATE_KEY);
// wallet.address is already EIP-55 checksummed

const cr = await fetch(`${API}/challenge?wallet=${wallet.address}`, {
  method: 'POST',
});
const { nonce, message } = await cr.json();

// sign the 'message' verbatim — same warning as §05
const signature = await wallet.signMessage(message);

const r = await fetch(`${API}/api/service/${SERVICE_ID}`, {
  headers: {
    'X-Wallet-Address': wallet.address,
    'X-Auth-Nonce': nonce,
    'X-Wallet-Signature': signature,
  },
});

const body = await r.json();
if (!r.ok) throw new Error(`${body.error}: ${body.detail}`);
return body;
```

The wallet that signs here must be the same one that registered for the
service and ran the attestation. If you're moving from browser usage
to server usage, you'll need to export the private key of the wallet
you used in the browser — or register again with a fresh wallet
dedicated to the server.

> Treat `SEPOLIA_PRIVATE_KEY` like any other secret. The wallet that
> registered for the service IS the credential for that subscription;
> leaking it leaks the subscription.

## 07 · Attestation lifetime + refresh

Default TTL: **1 hour**. Owner-settable on the marketplace contract,
bounded to [1 min, 24 h].

Within the TTL window, you can call the API as many times as you want
— only the wallet signature per call, no gas. After expiration, the
next call returns:

```http
HTTP 401
{
  "error": "no_fresh_attestation",
  "detail": "No on-chain attestation within the 3600-second window.
             Run verifyAndAttest first.",
  "wallet": "0x..."
}
```

To refresh: go back to `/developer/service/N` and click "run on-chain
attestation" again. Same two-tx flow as your first time. ~30 seconds
elapsed, ~600K gas total.

For high-frequency usage you can ask the provider (or the marketplace
owner) to bump the TTL closer to the 24-hour maximum to amortize the
refresh cost.

## 08 · Key rotation + storage

Rotation generates a new random key and replaces the ciphertext stored
against your subscription. Use it if you suspect your key was leaked,
or as routine hygiene.

Developer dashboard → pick a subscription → "rotate to a new random
key." Costs gas only, no fee. The old key stops working immediately;
the new key auto-fills + auto-saves locally.

Rotation **does not affect your wallet identity** — your subscription
history, your call count, and your provider-side logs all survive
cleanly. The provider sees the same wallet calling them; the comparison
just uses different ciphertext.

### Where your plaintext key lives

- **This browser's localStorage** — encrypted with a session key
  derived from a wallet signature. A passive disk dump or browser
  extension reading localStorage gets ciphertext only.
- **Wherever you backed it up** — password manager, secure notes, an
  encrypted file. You.

The plaintext does NOT live: at the provider, at the marketplace, on
the chain, in the relayer, or in Zama's KMS. Just the encrypting
browser + your backup.

## 09 · Revocation + recovery

A provider can revoke any subscription at any time. The next attestation
attempt against a revoked subscription reverts on-chain. Your existing
attestation (if still fresh) keeps working until the TTL expires — but
you won't be able to refresh.

To get re-access:

- Ask the provider to un-revoke (they have a per-subscription button on
  their dashboard). Sub-zero gas cost for them.
- Or: register again from scratch with a fresh wallet, paying the
  registration fee again. The provider can't prevent this (anyone can
  register), but they can revoke you again.

Note: there's no "developer-initiated unsubscribe" today. If you want
to stop using a service, just stop calling it. The fee is non-refundable.

## 10 · Error codes you might see

| Status | Error | What it means / what to do |
|---|---|---|
| 400 | `missing_headers` | Forgot one of the three auth headers. Send all three. |
| 400 | `bad_address` | Wallet header isn't a valid Ethereum address. Run it through `ethers.getAddress()` first. |
| 401 | `no_challenge` | Skipped `POST /challenge`. Fetch one first. |
| 401 | `nonce_mismatch` | You sent the wrong nonce. Re-fetch a challenge. |
| 401 | `challenge_expired` | 2-minute TTL elapsed. Re-fetch. |
| 401 | `signature_malformed` | Your signature isn't valid ECDSA hex. Probably mangled in transit, or you reconstructed it instead of using `signer.signMessage()`. Re-sign the unmodified `message` field. |
| 401 | `signature_mismatch` | You signed with a different wallet than the one in the header, the header isn't EIP-55 checksummed, or — most commonly — you rebuilt the message instead of signing the exact `message` string returned by `/challenge`. |
| 401 | `no_fresh_attestation` | Your attestation expired or doesn't exist. Run a new `verifyAndAttest` at `/developer/service/N`. |
| 401 | `attestation_invalid` | You attested but with the wrong key. Either you typed the key wrong or your provider's stored ciphertext is for a different subscription. |
| 401 | `service_mismatch` | Your latest attestation is for a different service. Go back to `/developer/service/N` for *this* service and attest there. |
| 502 | `gate_not_ready` | Provider's server hasn't completed bootstrap (usually transient). Retry in 10-30 seconds. |
| 502 | `contract_read_failed` | Provider's Sepolia RPC is degraded. Retry with exponential backoff; usually transient (Infura/Alchemy hiccup). |

## 11 · Cost model

| Action | Cost | Frequency |
|---|---|---|
| Register for a service | 0.0003 ETH + ~200K gas | Once per service you join |
| Run an attestation | ~600K gas total (2 txs) | Once per TTL window (default 1h) |
| Call the API | **No gas, no fee** | Unlimited within the TTL |
| Rotate your key | ~200K gas (no fee) | When you decide |

On Sepolia, gas is play money. If/when this moves to mainnet,
attestation refresh becomes the main recurring cost — bump the TTL to
24h to amortize.

## 13 · FAQ

**Do I need real ETH?**
No. Sepolia is a testnet. Faucets give you free Sepolia ETH for
development.

**Where is my plaintext key actually stored?**
Only in this browser's localStorage (encrypted with a session key
derived from a wallet signature) and wherever you backed it up.
Nothing else has a copy.

**Can I use the same wallet for multiple services?**
Yes. Each (wallet, service) pair has its own subscription with its own
key. One wallet can hold many subscriptions.

**Can I share a subscription across multiple machines?**
Yes. The plaintext API key is the random 32 bytes your browser
generated at registration — there's nothing wallet-derived about it.
Copy that hex string (from your one-time reveal or your backup) to
the new machine, use the same wallet for signing, and you're in. The
local key store on the new machine will be empty — paste the key in
manually on the use page, then save it locally if you want auto-fill.

Important distinction: the wallet signature *encrypts your
localStorage entry*; it does NOT generate the key itself. If your
wallet signature could regenerate your API key, anyone with your
wallet could too — and the whole privacy story would collapse. The
32 random bytes are the actual secret. The wallet just controls who
can read the encrypted local copy on a given device.

**What if I lose access to my wallet?**
You lose the subscription. The wallet IS the identity; no wallet, no
possession proof. Treat the wallet like any other credential.

**What if I lose the plaintext key but still have the wallet?**
Ask the provider to revoke your subscription and register again with
the same wallet. You'll generate a new key and pay the registration
fee again. (Or: rotate, if the old key isn't fully lost and the rotate
flow still works with your current attestation.)

**Does the provider see my requests?**
They see your wallet address, the time of the request, and your call
patterns — same as any API provider sees its customers. They don't
see your key (ever). They can build their own usage analytics keyed by
wallet.

**Can I see how providers track my usage?**
On-chain, yes: `getSubscription(serviceId, yourWallet)` returns a
`callCount` if the provider opted into the contract's public counter.
Off-chain, providers can track whatever they want; that's up to them.
